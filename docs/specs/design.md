# Bugfix 设计

> **主题：** 修复 Django + React + Electron 重构后的生产可用性回归
> **状态：** 草稿
> **最后更新：** 2026-03-31
> **对应文档：** `docs/specs/bugfix.md`

---

## 1. 设计目标

本设计只服务于本次 bugfix，目标是以最小安全改动恢复以下已声明行为：

1. Electron 生产包能够加载前端并进入 React 应用。
2. Electron 生产包能够在当前项目既有环境假设下拉起本地 Django 后端。
3. 地图边界在地图实例就绪后稳定绘制。
4. 路径规划失败时返回稳定的 JSON 错误契约。

本设计不引入新的产品能力，不重做构建体系，不扩大为“完整离线桌面化改造”。

---

## 2. 根因分析

### RC-001: 渲染资源路径与加载协议不匹配

- Webpack 输出面向站点根路径，适合 `http://host/`
- Electron 生产模式实际通过 `file://.../index.html` 打开页面
- 根路径资源引用在本地文件协议下失效

### RC-002: API 地址真相源错误

- 前端把 API 地址写死在构建期环境变量
- Electron 主进程没有参与后端编排
- 打包清单没有后端运行资源
- 结果是“桌面壳”只是包了一个前端静态页面

### RC-003: React effect 依赖遗漏

- 边界绘制依赖的不只是 `mapConfig`，还依赖地图实例和地图 SDK 就绪状态
- 代码把一次性数据依赖误写成时序依赖，导致 race condition

### RC-004: 上游异常边界不闭合

- `requests` 网络错误、超时、JSON 解析错误和高德业务错误属于不同异常源
- 目前只有高德业务错误被转换成结构化应用错误

## 3. 修复策略

### 3.1 前端资源加载策略

#### 方案

- 保留开发态 `webpack-dev-server`
- 生产构建改为输出相对静态资源地址
- `index.html` 中静态资源引用不得以 `/` 开头
- 仅在生产态调整资源寻址，不引入额外 runtime hack

#### 设计约束

- 不允许在 Electron 主进程里注入字符串替换 HTML
- 不允许通过注册自定义协议来掩盖本质上的路径错误，除非相对路径方案无法满足高德 JS 资源加载

### 3.2 Electron 生产态本地后端编排

#### 方案

Electron 主进程在生产态承担单一职责：

1. 确定本地 API 基地址
2. 启动 Django 子进程
3. 轮询一个轻量健康检查端点直到后端可用或超时失败
4. 通过 preload 向渲染进程注入运行时 API 基地址
5. 退出应用时回收子进程

#### 运行模式

- 开发态：
  - 继续加载 `http://localhost:3000`
  - API 默认指向外部 Django `http://localhost:8000/api`
- 生产态：
  - 加载本地 `dist/index.html`
  - API 基地址来自主进程注入，例如 `http://127.0.0.1:<runtime-port>/api`
  - Django 以 loopback 监听，仅供本机 Electron 调用

#### 后端打包范围

- 将 `backend/` 源码、`backend/requirements.txt` 和运行所需 Electron 启动脚本纳入打包资源
- 不在本次 bugfix 中引入新的 Python 打包器
- 支持范围明确限定为“本机存在可调用 Python 和可用 PostgreSQL”的环境

#### 失败策略

- 如果后端在限定时间内未就绪，生产态应 fail-fast
- 错误应显示为明确启动失败，而不是偷偷回退到外部 `localhost:8000`

### 3.3 运行时 API 配置单一真相源

#### 方案

- `src/api/client.js` 从单一配置入口读取 API 基地址
- 该入口优先使用 preload 注入的运行时值，开发态再回退到构建期环境变量

#### 约束

- 不允许同时维护“preload 常量 + env 常量 + 组件内临时拼接”三套来源
- 不允许按请求失败与否动态切换目标地址

### 3.4 地图边界绘制修复

#### 方案

- 将“地图已创建”抽象成明确可追踪的 ready 状态
- 边界绘制 effect 依赖 `mapConfig` 和地图实例就绪信号
- 当边界已存在时先移除再重绘，保持幂等

#### 非方案

- 不接受用 `setTimeout` 延迟绘制来赌时序
- 不接受在多个组件重复触发边界绘制来碰运气

### 3.5 路径规划错误契约修复

#### 方案

- 在 `AMapClient` 层统一把以下异常转换为应用层错误类型：
  - 网络异常
  - 超时
  - 非 JSON 响应
  - 高德业务失败响应
- 视图层继续只处理“参数错误”和“应用层上游错误”两类情况

#### 约束

- 不吞掉异常后返回空路线
- 不把所有异常都包装成同一个模糊文案

### 前端 / Electron

- `frontend/electron/main.js`
- `frontend/electron/preload.js`

### 后端

- `backend/routing/amap_client.py`
- `backend/routing/views.py`
- 如需健康检查复用现有接口，可评估是否复用 `config/map/`；若引入独立健康端点，必须保持最小化

## 5. 备选方案与取舍

| 方案 | 结论 | 原因 |
|:---|:---|:---|
| 仅把 `publicPath` 改为相对路径，不处理后端启动 | 放弃 | 只能修白屏，无法恢复桌面功能链路 |
| 生产态继续硬编码 `localhost:8000` | 放弃 | 把环境依赖继续暴露给最终用户，且与打包交付目标冲突 |
| 请求失败时自动尝试多个 API 地址 | 放弃 | 会制造隐藏分支和不可诊断行为，属于兜底式修补 |
| 用 `setTimeout` 或多次 effect 触发修地图边界 | 放弃 | 治标不治本，时序仍不确定 |
| 用自定义协议重写所有资源路径 | 备选 | 只有在相对路径方案被高德资源加载限制阻断时才考虑 |

## 6. 验证策略

### V-001: 白屏回归证明

- 构建后检查 `dist/index.html` 中 JS 和 favicon 引用为相对路径
- 生产模式启动 Electron 时确认 React 根节点实际渲染，而非空白窗口

### V-002: 后端编排证明

- 生产模式启动时，Electron 主进程可在限定时间内拉起 Django
- 渲染进程通过运行时注入的 API 基地址成功请求至少一个后端端点
- 退出应用后 Django 子进程被清理

### V-003: 地图边界修复证明

- 地图初始化完成后边界一定存在
- 搜索、筛选、路线绘制不会导致边界永久消失

### V-004: 错误契约证明

- 模拟高德网络失败或非法响应时，接口返回 JSON `{error: ...}` 而不是 Django HTML 500

---

## 7. 实施边界

1. 允许增加最小必要的 Electron 启动辅助代码和验证脚本。
2. 不允许顺手重写 UI、改造状态管理或更换构建工具。
3. 若在实施中发现“需要打包 Python 解释器或 PostgreSQL 才能满足目标”，必须先回写规范再继续，不得现场扩需求。
